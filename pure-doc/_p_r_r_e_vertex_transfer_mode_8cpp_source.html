33  GeForce 2 cards (and probably GeForce 4MX) only supported 16-bit indexing in HW, so
34  if you went over 64Ki vertices then it would have to do the indexing in software.
35 
36  Some manual say that glDrawElements()' type parameter must be GL_UNSIGNED_BYTE or GL_UNSIGNED_SHORT, while some allow GL_UNSIGNED_INT too.
37  Depends on implementation. This also means that we must use the smallest type given to glDrawElements() not only because of
38  performance considerations but also in order to make the function work on older implementations.
39 
40  glDrawRangeElements() indices and vertices limits:
41  http://www.opengl.org/registry/specs/EXT/draw_range_elements.txt
42  The specification of glDrawElements does not allow optimal performance for some OpenGL implementations, however.
43  In particular, it has no restrictions on the number of indices given, the number of unique vertices referenced nor a
44  direct indication of the set of unique vertices referenced by the given indices. This forces some OpenGL
45  implementations to walk the index data given, building up a separate list of unique vertex references for later use
46  in the pipeline. Additionally, since some OpenGL implementations have internal limitations on how many vertices they can
47  deal with simultaneously the unbounded nature of glDrawElements requires the implementation to be prepared to segment the
48  input data and do multiple passes. These preprocessing steps can consume a significant amount of time.
49  glDrawRangeElementsEXT is designed to avoid the preprocessing steps which may be necessary for glDrawElements.
50  As such it does not have the flexibility of glDrawElements but it is sufficiently functional for a large class of applications
51  to benefit from its use. glDrawRangeElementsEXT enhances glDrawElements in two ways:
52  1. The set of unique vertices referenced by the indices is explicitly indicated via the start and end parameters, removing the
53  necessity to determine this through examination of the index data. The implementation is given a contiguous chunk of vertex
54  data that it can immediately begin streaming through the vertex pipeline.
55  2. Recommended limits on the amount of data to be processed can be indicated by the implementation through GL_MAX_ELEMENTS_VERTICES_EXT
56  and GL_MAX_ELEMENTS_INDICES_EXT. If an application respects these limits it removes the need to split the incoming data into multiple
57  chunks since the maximums can be set to the optimal values for the implementation to handle in one pass.
58 
59  http://www.opengl.org/discussion_boards/showthread.php/146440-glDrawRangeElements-GL_MAX_ELEMENTS_VERTICES-INDICES
60  However the performance effects of these caches has to do with the re-use of indices and
61  the relative locality of vertex array contents. These performance effects aren't really
62  related to the number of vertices actually processed by a particular glDrawRangeElements as
63  the limits above are specified to describe.
64  It's tempting to "re-interpret" these limits to be more "relevant" to how modern GPUs work, but
65  that is not what the limits are specified to mean.
66 
67  http://stackoverflow.com/questions/7549991/questions-about-gldrawrangeelements
68  Now that we know how to use glDrawElements, let's talk about glDrawRangeElements.
69  When normally using glDrawElements, you specify a location in the source index array to pull from.
70  The indices and count parameters tell OpenGL where to find the indices. But the actual indices pulled from
71  this array could be anywhere within the boundaries of the source vertex array indices.
72  glDrawRangeElements allows you to give OpenGL a range (half-open again) of vertex index values.
73  What you are saying is that the indices it gets during this draw call will not exceed that range (half-open).
74  This could allow the driver to perform useful optimizations. The start value should be the lowest index value
75  that will be gotten from the index array, and the end should be the highest (+1 because it's half-open).
76  It should not simply be the index of the first and last vertices.
77 
78  http://www.songho.ca/opengl/gl_vertexarray.html
79  The additional parameters in glDrawRangeElements() are start and end index, then OpenGL prefetches a limited
80  amount of vertices from these values: end - start + 1. And the values in index array must lie in between
81  start and end index. Note that not all vertices in range (start, end) must be referenced. But, if you specify a
82  sparsely used range, it causes unnecessary process for many unused vertices in that range.

197 TPRREbool PRREVertexTransfer::PRREVertexTransferImpl::setVertexTransferMode(TPRRE_VERTEX_TRANSFER_MODE vtrans)
198 {
199  _pOwner->getManagedConsole().OLnOI("PRREVertexTransfer::setVertexTransferMode(%u)", vtrans);
200 
201  if ( _pOwner->isLevel2() )
202  {
203  if ( !_pOwner->getManager() )
204  {
205  // being level-2 without a manager is unusual, but can be achieved when:
206  // either user invokes Detach() for a submesh of a complete level-1 mesh,
207  // or when we are still creating a mesh and it is not yet attached or it does not have any children.
208  // any case, we should IGNORE execution of this function.
209  _pOwner->getManagedConsole().EOLnOO("setVertexTransferMode(%u) of submesh called WITHOUT existing level-1 parent mesh, rejecting!", vtrans);
210  return false;
211  }
212 
213  // submesh must reject if its setVertexTransferMode() was not called by its parent level-1 mesh but someone else from outside ...
214  // note: our manager in this case is a level-1 VertexTransfer (Mesh3D) mesh
215  if ( !((PRREVertexTransfer*)_pOwner->getManager())->pImpl->bParentInitiatedOperation )
216  {
217  _pOwner->getManagedConsole().EOLnOO("setVertexTransferMode(%u) of submesh called outside of its level-1 parent mesh, rejecting!", vtrans);
218  return false;
219  }
220  }
221 
222  // Note: we don't check if the selected transfer mode is already selected.
223  // Reason: selecting an already selected transfer mode is a way to recreate buffers / refresh geometry on server side.
224  // Modify geometry in system RAM and simply issue setVertexTransferMode( getVertexTransferMode() ) to apply changes.
225 
226  if ( PRREVertexTransfer::isVertexReferencingIndexed(vertexTransferMode) &&
227  !PRREVertexTransfer::isVertexReferencingIndexed(vtrans) )
228  {
229  if ( !isSwitchFromIndexedAllowed() )
230  {
231  _pOwner->getManagedConsole().EOLnOO("ERROR: failed to switch from indexed to direct referencing mode!");
232  return false;
233  }
234  }
235 
236  if ( !PRREVertexTransfer::isVertexTransferModeSelectable(vtrans) )
237  {
238  _pOwner->getManagedConsole().EOLnOO("ERROR: given mode %u is unavailable!", vtrans);
239  return false;
240  }
241 
242  if ( _pOwner->isLevel2() )
243  {
244  FreeGLresources(); // now it's safe to delete GL resources
245  vertexTransferMode = 0;
246 
247  if ( BIT_READ(vtrans, PRRE_VT_SVA_BIT) )
248  {
249  /* we want VBO */
250  if ( !compileIntoVertexBufferObjects(
251  PRREVertexTransfer::isVertexReferencingIndexed(vtrans),
252  PRREVertexTransfer::isVertexModifyingDynamic(vtrans) ) )
253  {
254  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
255  _pOwner->getManagedConsole().EOLnOO("ERROR: compileIntoVertexBufferObjects() failed!");
256  return false;
257  }
258  }
259  else
260  {
261  /* check if we want display list? */
262  if ( (vtrans == (PRRE_VMOD_STATIC | PRRE_VREF_DIRECT)) ||
263  (vtrans == (PRRE_VMOD_STATIC | PRRE_VREF_INDEXED)) )
264  {
265  if ( !compileIntoDisplayList(PRREVertexTransfer::isVertexReferencingIndexed(vtrans)) )
266  {
267  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
268  _pOwner->getManagedConsole().EOLnOO("ERROR: compileIntoDisplayList() failed!");
269  return false;
270  }
271  }
272  }
273 
274  /* no other transfer mode needs further preparation */
275  vertexTransferMode = vtrans;
276  }
277  else
278  {
279  bParentInitiatedOperation = true;
280  _pOwner->getManagedConsole().OLn("Ok, applying for subobjects ...");
281  for (TPRREint i = 0; i < _pOwner->getCount(); i++)
282  {
283  if ( !((PRREVertexTransfer*)(_pOwner->getAttachedAt)(i))->setVertexTransferMode( vtrans ) )
284  {
285  _pOwner->getManagedConsole().EOLn("Failed a setVertexTransferMode(%u) for subobject %d!", vtrans, i);
286  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
287  vertexTransferMode = 0;
288  bParentInitiatedOperation = false;
289  _pOwner->getManagedConsole().OO();
290  return false;
291  }
292  }
293  _pOwner->getManagedConsole().OLn("Done!");
294  bParentInitiatedOperation = false;
295  vertexTransferMode = vtrans;
296  }
297 
298  _pOwner->getManagedConsole().OO();
299  return true;
300 } // setVertexTransferMode()

370 TPRREuint PRREVertexTransfer::PRREVertexTransferImpl::transferVertices()
371 {
372  assert(_pOwner->isLevel2());
373  if ( !_pOwner->isLevel2() )
374  {
375  _pOwner->getManagedConsole().EOLn("ERROR: PRREVertexTransfer::%s() invoked for level-1!", __FUNCTION__);
376  return 0;
377  }
378 
379  /* currently not supporting any vendor-specific mode */
380  if ( BITF_READ(getVertexTransferMode(),PRRE_VT_VENDOR_BITS,3) != 0 )
381  {
382  _pOwner->getManagedConsole().EOLn("ERROR: PRREVertexTransfer::%s() unsupported vendor-specific mode!", __FUNCTION__);
383  return 0;
384  }
385 
386  nLastTransferredVertices = _pOwner->getVertexIndicesCount();
387  nLastTransferredTriangles = _pOwner->getTriangleCount();
388  // increasing the number in the parent too, however it must be parent's logic to zero it out before iterating over its children again
389  ((PRREVertexTransfer*)(_pOwner->getManager()))->pImpl->nLastTransferredVertices += nLastTransferredVertices;
390  ((PRREVertexTransfer*)(_pOwner->getManager()))->pImpl->nLastTransferredTriangles += nLastTransferredTriangles;
391 
392  // this function doesn't check for GL errors since this is also part of rendering,
393  // renderer should check for errors after each frame
394 
395  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_VA_BIT) == 0u )
396  {
397  /* there is no array for geometry, so we either invoke a display list or pass vertices 1-by-1 */
398 
399  if ( PRREVertexTransfer::isVertexModifyingDynamic( getVertexTransferMode() ) )
400  {
401  // PRRE_VT_DYN_..._1_BY_1
402  ProcessGeometry( PRREVertexTransfer::isVertexReferencingIndexed( getVertexTransferMode() ) );
403  }
404  else
405  {
406  // PRRE_VT_STA_..._DL
407  assert(nDispList > 0);
408  glCallList( nDispList );
409  }
410 
411  return nLastTransferredVertices;
412  } // PRRE_VT_VA_BIT == 0u
413 
414  /* vertex array either in client or server */
415  const bool bServer = BIT_READ(getVertexTransferMode(), PRRE_VT_SVA_BIT) == 1u;
416 
417  SetArrayPointers( bServer );
418 
419  // 1 big benefit of using CVA is when we want to do multitexturing with only
420  // 1 TMU, meaning that we need to do it in at least 2 passes, specifying
421  // vertex geometry twice! However even TNT had 2 TMUs, so I'm not planning to do this improvement.
422  // Another big benefit is when we have shared vertices, in indexed rendering
423  // all vertices are transformed if lock is enabled, thus shared vertices
424  // can be used multiple times EVEN THOUGH THEY ARE TRANSFORMED ONLY ONCE!
425  // Too bad we currently create redundant vertex geometry when loading OBJ mesh,
426  // however we have backlog item for that ...
427  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_CVA_BIT) == 1u )
428  glLockArraysEXT(0, _pOwner->getVerticesCount());
429 
430  // Static and dynamic vmods are not differentiated here but when creating the arrays.
431  // That is why we don't check for vmod in this code.
432 
433  if ( PRREVertexTransfer::isVertexReferencingIndexed( getVertexTransferMode() ) )
434  {
435  assert(_pOwner->getVertexIndicesCount() != 0);
436  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_RNG_BIT) )
437  {
438  // TODO: should call this in loop, do multiple batches based on the implementation-dependent values
439  // (GL_MAX_ELEMENTS_VERTICES_EXT, GL_MAX_ELEMENTS_INDICES_EXT).
440  glDrawRangeElementsEXT(
441  getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()),
442  _pOwner->getMinVertexIndex(false), _pOwner->getMaxVertexIndex(false),
443  _pOwner->getVertexIndicesCount(), _pOwner->getVertexIndicesType(false), bServer ? NULL : _pOwner->getVertexIndices(false) );
444  }
445  else
446  {
447  glDrawElements(
448  getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()),
449  _pOwner->getVertexIndicesCount(), _pOwner->getVertexIndicesType(false), bServer ? NULL : _pOwner->getVertexIndices(false) );
450  }
451  }
452  else
453  {
454  assert(_pOwner->getVerticesCount() != 0);
455  glDrawArrays( getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()), 0, _pOwner->getVerticesCount() );
456  }
457 
458  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_CVA_BIT) == 1u )
459  glUnlockArraysEXT();
460 
461  ResetArrayPointers( bServer );
462 
463  return nLastTransferredVertices;
464 }

997 TPRRE_VERTEX_TRANSFER_MODE PRREVertexTransfer::selectVertexTransferMode(
998  TPRRE_VERTEX_MODIFYING_HABIT vmod,
999  TPRRE_VERTEX_REFERENCING_MODE vref,
1000  TPRREbool bForceUseClientMemory)
1001 {
1002  TPRRE_VERTEX_TRANSFER_MODE ret = 0;
1003 
1004  if ( vref == PRRE_VREF_INDEXED )
1005  ret = ret | PRRE_VREF_INDEXED;
1006 
1007  // dont go further if CPU is rendering
1008  if ( !PRREhwInfo::get().getVideo().isAcceleratorDetected() )
1009  return ret | PRRE_VMOD_DYNAMIC;
1010 
1011  if ( vmod == PRRE_VMOD_DYNAMIC )
1012  ret = ret | PRRE_VMOD_DYNAMIC;
1013 
1014  if ( PRREhwInfo::get().getVideo().isDrawRangeElementsSupported() )
1015  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_RNG_BIT);
1016 
1017  if ( PRREhwInfo::get().getVideo().isCompiledVertexArraySupported() )
1018  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_CVA_BIT);
1019 
1020  // bForceUseClientMemory is considered only if modifying habit is dynamic
1021  if ( bForceUseClientMemory && (vmod == PRRE_VMOD_DYNAMIC) )
1022  return ret | BIT(PRRE_VT_VA_BIT); // client-side should use an array, even if none of RNG and CVA bits are available, to avoid selecting immediate mode
1023 
1024  // At this point we want to use host/server memory, but it could be still either client- or server-side array, or display list or immediate mode, so
1025  // we should set PRRE_VT_VA_BIT only in case of some arraying.
1026 
1027  // if general VBO is supported, return with that immediately as we don't want to stick to any vendor-specific magical stuff.
1028  if ( PRREhwInfo::get().getVideo().isVertexBufferObjectSupported() )
1029  return ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT);
1030 
1031  if ( PRREhwInfo::get().getVideo().isATIVertexArrayObjectSupported() )
1032  {
1033  if ( PRREhwInfo::get().getVideo().isATIElementArraySupported() )
1034  ret = ret | BIT(PRRE_VT_RNG_BIT) | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_ATI_2,PRRE_VT_VENDOR_BITS,3);
1035  else
1036  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_ATI_1,PRRE_VT_VENDOR_BITS,3);
1037  }
1038  else
1039  {
1040  if ( PRREhwInfo::get().getVideo().isNVVertexArrayRangeSupported() )
1041  {
1042  if ( PRREhwInfo::get().getVideo().isNVElementArraySupported() )
1043  ret = ret | BIT(PRRE_VT_RNG_BIT) | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_NVIDIA_2,PRRE_VT_VENDOR_BITS,3);
1044  else
1045  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_NVIDIA_1,PRRE_VT_VENDOR_BITS,3);
1046  }
1047  }
1048 
1049  // at this point, if VA_BIT is not set, the selected mode is either display list or immediate mode, depending on vmod
1050 
1051  return ret;
1052 } // selectVertexTransferMode()

1216 TPRREuint PRREVertexTransfer::getUsedVideoMemory() const
1217 {
1218  // level-2 objects' vertex transfer mode might or might not be set, but anyway, always the transfer mode of their owner level-1 object counts!
1219  const TPRRE_VERTEX_TRANSFER_MODE& transferMode =
1220  isLevel1() ?
1221  getVertexTransferMode() :
1222  ((PRREVertexTransfer*)getManager())->getVertexTransferMode();
1223 
1224  if ( !PRREVertexTransfer::isVideoMemoryUsed(transferMode) )
1225  return 0;
1226 
1227  TPRREuint sumSubMeshMemoryUsage = 0;
1228 
1229  if ( isLevel1() )
1230  {
1231  for (TPRREint i = 0; i < getSize(); i++)
1232  {
1233  const PRREVertexTransfer* const subobj = (PRREVertexTransfer*) getAttachedAt(i);
1234  if ( subobj == PGENULL )
1235  continue;
1236  sumSubMeshMemoryUsage += subobj->getUsedVideoMemory();
1237  }
1238  return sumSubMeshMemoryUsage;
1239  }
1240 
1241  // at this point this is a level-2, we keep using sumSubMeshMemoryUsage for its mem usage
1242 
1243  // VBO or display list, it doesn't matter, we always calculate with vertices, normals, colors, and 1 or 2 texture uvw arrays,
1244  // since these are together always specified for both VBO and display lists.
1245  sumSubMeshMemoryUsage = getVerticesCount() * sizeof(TXYZ) * 2 /* *2 because of we have normals too */ +
1246  getMaterial(false).getColorsCount() * sizeof(TRGBAFLOAT) +
1247  getMaterial(false).getTexcoordsCount() * sizeof(TUVW);
1248 
1249  if ( getMaterial(false).getTexcoords(1) != PGENULL )
1250  {
1251  sumSubMeshMemoryUsage += getMaterial(false).getTexcoordsCount(1) * sizeof(TUVW);
1252  }
1253 
1254  // In case of VBO, we also add index (element) array if it is indexed rendering
1255  if ( (BIT_READ(transferMode, PRRE_VT_SVA_BIT) == 1u) )
1256  {
1257  if ( pImpl->nIndicesVBO != 0 )
1258  {
1259  sumSubMeshMemoryUsage += getVertexIndicesCount() * PRREGLsnippets::getSizeofIndexType(getVertexIndicesType());
1260  }
1261  }
1262 
1263  return sumSubMeshMemoryUsage;
1264 } // getUsedVideoMemory()
