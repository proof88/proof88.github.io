29  ***************************
30 
31  GeForce 2 cards (and probably GeForce 4MX) only supported 16-bit indexing in HW, so
32  if you went over 64Ki vertices then it would have to do the indexing in software.
33 
34  Some manual say that glDrawElements()' type parameter must be GL_UNSIGNED_BYTE or GL_UNSIGNED_SHORT, while some allow GL_UNSIGNED_INT too.
35  Depends on implementation. This also means that we must use the smallest type given to glDrawElements() not only because of
36  performance considerations but also in order to make the function work on older implementations.
37 
38  glDrawRangeElements() indices and vertices limits:
39  http://www.opengl.org/registry/specs/EXT/draw_range_elements.txt
40  The specification of glDrawElements does not allow optimal performance for some OpenGL implementations, however.
41  In particular, it has no restrictions on the number of indices given, the number of unique vertices referenced nor a
42  direct indication of the set of unique vertices referenced by the given indices. This forces some OpenGL
43  implementations to walk the index data given, building up a separate list of unique vertex references for later use
44  in the pipeline. Additionally, since some OpenGL implementations have internal limitations on how many vertices they can
45  deal with simultaneously the unbounded nature of glDrawElements requires the implementation to be prepared to segment the
46  input data and do multiple passes. These preprocessing steps can consume a significant amount of time.
47  glDrawRangeElementsEXT is designed to avoid the preprocessing steps which may be necessary for glDrawElements.
48  As such it does not have the flexibility of glDrawElements but it is sufficiently functional for a large class of applications
49  to benefit from its use. glDrawRangeElementsEXT enhances glDrawElements in two ways:
50  1. The set of unique vertices referenced by the indices is explicitly indicated via the start and end parameters, removing the
51  necessity to determine this through examination of the index data. The implementation is given a contiguous chunk of vertex
52  data that it can immediately begin streaming through the vertex pipeline.
53  2. Recommended limits on the amount of data to be processed can be indicated by the implementation through GL_MAX_ELEMENTS_VERTICES_EXT
54  and GL_MAX_ELEMENTS_INDICES_EXT. If an application respects these limits it removes the need to split the incoming data into multiple
55  chunks since the maximums can be set to the optimal values for the implementation to handle in one pass.
56 
57  http://www.opengl.org/discussion_boards/showthread.php/146440-glDrawRangeElements-GL_MAX_ELEMENTS_VERTICES-INDICES
58  However the performance effects of these caches has to do with the re-use of indices and
59  the relative locality of vertex array contents. These performance effects aren't really
60  related to the number of vertices actually processed by a particular glDrawRangeElements as
61  the limits above are specified to describe.
62  It's tempting to "re-interpret" these limits to be more "relevant" to how modern GPUs work, but
63  that is not what the limits are specified to mean.
64 
65  http://stackoverflow.com/questions/7549991/questions-about-gldrawrangeelements
66  Now that we know how to use glDrawElements, let's talk about glDrawRangeElements.
67  When normally using glDrawElements, you specify a location in the source index array to pull from.
68  The indices and count parameters tell OpenGL where to find the indices. But the actual indices pulled from
69  this array could be anywhere within the boundaries of the source vertex array indices.
70  glDrawRangeElements allows you to give OpenGL a range (half-open again) of vertex index values.
71  What you are saying is that the indices it gets during this draw call will not exceed that range (half-open).
72  This could allow the driver to perform useful optimizations. The start value should be the lowest index value
73  that will be gotten from the index array, and the end should be the highest (+1 because it's half-open).
74  It should not simply be the index of the first and last vertices.
75 
76  http://www.songho.ca/opengl/gl_vertexarray.html
77  The additional parameters in glDrawRangeElements() are start and end index, then OpenGL prefetches a limited
78  amount of vertices from these values: end - start + 1. And the values in index array must lie in between
79  start and end index. Note that not all vertices in range (start, end) must be referenced. But, if you specify a
80  sparsely used range, it causes unnecessary process for many unused vertices in that range.
81

85 
86  Compiled vertex array: lock-unlock, but only 1 can be locked at a time, so the only use of this may be
87  when the same geometry is rendered multiple times like multipass blending if no multitexturing available.
88  Some performance gain may be observed also when using indices (drawElements) because shared vertices will be
89  transformed and lit only once. This also means there is no such post-TnL cache optimisation with standard drawElements.
90 
91  Early vendor-specific "VBO"s:
92  - GeForce2: VAR: NV_vertex_array_range, NV_vertex_array_range2
93  VRAM or AGP memory. Write is slow. Write is fast only if Fast Writes is supported.
94  Rule: allocate static geometry in VRAM, allocate AGP mem for dynamic geometry.
95  Modifying AGP memory is slow, so first work in system memory, then block-copy to AGP memory.
96  - VAO: ATI_vertex_array_object, ATI_map_object_buffer, ATI_element_array (like ELEMENT_ARRAY_BUFFER_ARB target).
97  Can't be AGP mem, only VRAM.

100  - ATI_vertex_array_object: STATIC_ATI, DYNAMIC_ATI
101  - NV_vertex_array_range: readFreq, writeFreq, priority
102  - ARB_vertex_buffer_object: STATIC, STREAM, DYNAMIC; DRAW, READ, COPY
103 
104 
105  BufferObject: a GPU allocated memory buffer
106  Vertex Buffer Object: a BufferObject containing vertices informations (colors, position, custom data used by a shader, ...)
107  Pixel Buffer Object: a BufferObject containing pixel or texel informations. Mainly used to upload textures.
108  Element Buffer Object: a BufferObject containing indices (used by glDrawElements).
109  Vertex Array: memory used by gl*Pointer call.
110  Might be host memory or a Vertex Buffer Object if it is bound using glBindBuffer command with GL_ARRAY_BUFFER.
111  Element Array: memory used by glDrawElements call.
112  Might be host memory or an Element Buffer Object if it is bound using glBindBuffer command with GL_ELEMENT_ARRAY_BUFFER.

207 {
208  _pOwner->getManagedConsole().OLnOI("PRREVertexTransfer::setVertexTransferMode()");
209 
210  if ( _pOwner->isLevel2() )
211  {
212  if ( !_pOwner->getManager() )
213  {
214  // being level-2 without a manager is unusual, but can be achieved when:
215  // either user invokes Detach() for a submesh of a complete level-1 mesh,
216  // or when we are still creating a mesh and it is not yet attached or it does not have any children.
217  // any case, we should IGNORE execution of this function.
218  _pOwner->getManagedConsole().EOLnOO("setVertexTransferMode() of submesh called WITHOUT existing level-1 parent mesh, rejecting!");
219  return false;
220  }
221

223  // note: our manager in this case is a level-1 VertexTransfer (Mesh3D) mesh
224  if ( !((PRREVertexTransfer*)_pOwner->getManager())->pImpl->bParentInitiatedOperation )
225  {
226  _pOwner->getManagedConsole().EOLnOO("setVertexTransferMode() of submesh called outside of its level-1 parent mesh, rejecting!");
227  return false;
228  }
229  }
230 
231  // Note: we don't check if the selected transfer mode is already selected.
232  // Reason: selecting an already selected transfer mode is a way to recreate buffers / refresh geometry on server side.
233  // Modify geometry in system RAM and simply issue setVertexTransferMode( getVertexTransferMode() ) to apply changes.
234 
235  if ( PRREVertexTransfer::isVertexReferencingIndexed(vertexTransferMode) &&
236  !PRREVertexTransfer::isVertexReferencingIndexed(vtrans) )
237  {
238  if ( !isSwitchFromIndexedAllowed() )
239  {
240  _pOwner->getManagedConsole().EOLnOO("ERROR: failed to switch from indexed to direct referencing mode!");
241  return false;
242  }
243  }
244 
245  if ( !PRREVertexTransfer::isVertexTransferModeSelectable(vtrans) )
246  {
247  _pOwner->getManagedConsole().EOLnOO("ERROR: given mode %d is unavailable!", (int)vtrans);
248  return false;
249  }
250 
251  if ( _pOwner->isLevel2() )
252  {
253  FreeGLresources(); // now it's safe to delete GL resources
254  vertexTransferMode = 0;
255 
256  if ( BIT_READ(vtrans, PRRE_VT_SVA_BIT) )
257  {
258  /* we want VBO */
259  if ( !compileIntoVertexBufferObjects(
260  PRREVertexTransfer::isVertexReferencingIndexed(vtrans),
261  PRREVertexTransfer::isVertexModifyingDynamic(vtrans) ) )
262  {
263  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
264  _pOwner->getManagedConsole().EOLnOO("ERROR: compileIntoVertexBufferObjects() failed!");
265  return false;
266  }
267  }
268  else
269  {
270  /* check if we want display list? */
271  if ( (vtrans == (PRRE_VMOD_STATIC | PRRE_VREF_DIRECT)) ||
272  (vtrans == (PRRE_VMOD_STATIC | PRRE_VREF_INDEXED)) )
273  {
274  if ( !compileIntoDisplayList(PRREVertexTransfer::isVertexReferencingIndexed(vtrans)) )
275  {
276  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
277  _pOwner->getManagedConsole().EOLnOO("ERROR: compileIntoDisplayList() failed!");
278  return false;
279  }
280  }
281  }
282 
283  /* no other transfer mode needs further preparation */
284  vertexTransferMode = vtrans;
285  }
286  else
287  {
288  bParentInitiatedOperation = true;
289  _pOwner->getManagedConsole().OLn("Ok, applying for subobjects ...");
290  for (TPRREint i = 0; i < _pOwner->getCount(); i++)
291  {
292  if ( !((PRREVertexTransfer*)(_pOwner->getAttachedAt)(i))->setVertexTransferMode( vtrans ) )
293  {
294  _pOwner->getManagedConsole().EOLn("Failed a setVertexTransferMode() for subobject %d!", i);
295  FreeGLresources(); // we cannot know exactly what was allocated and what no, make sure we don't keep anything!
296  vertexTransferMode = 0;
297  bParentInitiatedOperation = false;
298  _pOwner->getManagedConsole().OO();
299  return false;
300  }
301  }
302  _pOwner->getManagedConsole().OLn("Done!");
303  bParentInitiatedOperation = false;
304  vertexTransferMode = vtrans;
305  }
306 
307  _pOwner->getManagedConsole().OO();
308  return true;
309 } // setVertexTransferMode()

364 void PRREVertexTransfer::PRREVertexTransferImpl::TransferVertices()
365 {
366  // this function doesn't check for GL errors since this is also part of rendering,
367  // renderer should check for errors after each frame
368 
369  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_VA_BIT) == 0u )
370  {
371  /* there is no array for geometry, so we either invoke a display list or pass vertices 1-by-1 */
372 
373  if ( PRREVertexTransfer::isVertexModifyingDynamic( getVertexTransferMode() ) )
374  {
375  // PRRE_VT_DYN_..._1_BY_1
376  ProcessGeometry( PRREVertexTransfer::isVertexReferencingIndexed( getVertexTransferMode() ) );
377  }
378  else
379  {
380  // PRRE_VT_STA_..._DL
381  assert(nDispList > 0);
382  glCallList( nDispList );
383  }
384 
385  return;
386  } // PRRE_VT_VA_BIT == 0u
387 
388  /* vertex array either in client or server */
389  const bool bServer = BIT_READ(getVertexTransferMode(), PRRE_VT_SVA_BIT) == 1u;
390 
391  SetArrayPointers( bServer );
392 
393  // 1 big benefit of using CVA is when we want to do multitexturing with only
394  // 1 TMU, meaning that we need to do it in at least 2 passes, specifying
395  // vertex geometry twice! However even TNT had 2 TMUs, so I'm not planning to do this improvement.
396  // Another big benefit is when we have shared vertices, in indexed rendering
397  // all vertices are transformed if lock is enabled, thus shared vertices
398  // can be used multiple times EVEN THOUGH THEY ARE TRANSFORMED ONLY ONCE!
399  // Too bad we currently create redundant vertex geometry when loading OBJ mesh,
400  // however we have backlog item for that ...
401  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_CVA_BIT) == 1u )
402  glLockArraysEXT(0, _pOwner->getVerticesCount(false));
403 
404  // Static and dynamic vmods are not differentiated here but when creating the arrays.
405  // That is why we don't check for vmod in this code.
406 
407  if ( PRREVertexTransfer::isVertexReferencingIndexed( getVertexTransferMode() ) )
408  {
409  assert(_pOwner->getVertexIndicesCount(false) != 0);
410  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_RNG_BIT) )
411  {
412  // TODO: should call this in loop, do multiple batches based on the implementation-dependent values
413  // (GL_MAX_ELEMENTS_VERTICES_EXT, GL_MAX_ELEMENTS_INDICES_EXT).
414  glDrawRangeElementsEXT(
415  getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()),
416  _pOwner->getMinVertexIndex(false), _pOwner->getMaxVertexIndex(false),
417  _pOwner->getVertexIndicesCount(false), _pOwner->getVertexIndicesType(false), bServer ? NULL : _pOwner->getVertexIndices(false) );
418  }
419  else
420  {
421  glDrawElements(
422  getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()),
423  _pOwner->getVertexIndicesCount(false), _pOwner->getVertexIndicesType(false), bServer ? NULL : _pOwner->getVertexIndices(false) );
424  }
425  }
426  else
427  {
428  assert(_pOwner->getVerticesCount(false) != 0);
429  glDrawArrays( getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()), 0, _pOwner->getVerticesCount(false) );
430  }
431 
432  if ( BIT_READ(getVertexTransferMode(), PRRE_VT_CVA_BIT) == 1u )
433  glUnlockArraysEXT();
434 
435  ResetArrayPointers( bServer );
436 }

483 {
484  /* Make sure we don't call GL functions when no accelerator is present */
485  if ( !PRREhwInfo::get().getVideo().isAcceleratorDetected() )
486  return;
487 
488  //const TRGBAFLOAT* const pColors = _pOwner->getMaterial(false).getColors(0);
489  //const void* const pColorIndices = _pOwner->getMaterial(false).getColorIndices(0);
490  const TUVW* const pTexcoords = _pOwner->getMaterial(false).getTexcoords(0);
491  // following can be NULL if material system supports only 1 level
492  const TUVW* const pTexcoords2 = _pOwner->getMaterial(false).getTexcoords(1);
493  //const void* const pTexcoordIndices2 = _pOwner->getMaterial(false).getTexcoordIndices(1);
494 
495  glBegin( getGLprimitiveFromPRREprimitive(_pOwner->getPrimitiveFormat()) );
496  for (TPRREuint i = 0; i < _pOwner->getVertexIndicesCount(false); i++)
497  {
498  const TXYZ& vertex = indexed ? _pOwner->getVertices(false)[ _pOwner->getVertexIndex(i) ] : _pOwner->getVertices(false)[i];
499  const TXYZ& normal = indexed ? _pOwner->getNormals(false)[ _pOwner->getVertexIndex(i) ] : _pOwner->getNormals(false)[i];
500  //const TRGBAFLOAT& color = indexed ? pColors[ _pOwner->getVertexIndex(pColorIndices, i) ] : pColors[i];

502

504  {
505  if ( _pOwner->getMaterial(false).isMultiTextured() )
506  {
507  assert(pTexcoords2 != NULL);
508  const TUVW& uvw2 = indexed ? pTexcoords2[ _pOwner->getVertexIndex(i) ] : pTexcoords2[i];

515  glMultiTexCoord2fARB(GL_TEXTURE0_ARB, uvw.u, uvw.v);
516  }
517  }
518  else
519  {
520  if ( _pOwner->getMaterial(false).getTexture() != PGENULL )
521  glTexCoord2f(uvw.u, uvw.v);
522  }
523 
524  // per-vertex color is disabled for compatibility with tmcsgfxlib wrapper, reason somewhere in setarraypointers()

526 
527  // in the past, we sent normal data only if bAffectedLights was true

529  // anyway I've commented out this condition since after refactor, VertexTransfer does not know such thing like bAffectedByLights.
530  //if ( bAffectedByLights )
531  glNormal3f(normal.x, normal.y, normal.z);

533  } // for k

535 } // ProcessGeometry()

957 TPRRE_VERTEX_TRANSFER_MODE PRREVertexTransfer::selectVertexTransferMode(
958  TPRRE_VERTEX_MODIFYING_HABIT vmod,
959  TPRRE_VERTEX_REFERENCING_MODE vref,
960  TPRREbool bForceUseClientMemory)
961 {
962  TPRRE_VERTEX_TRANSFER_MODE ret = 0;
963 
964  if ( vref == PRRE_VREF_INDEXED )
965  ret = ret | PRRE_VREF_INDEXED;
966 
967  // dont go further if CPU is rendering
968  if ( !PRREhwInfo::get().getVideo().isAcceleratorDetected() )
969  return ret | PRRE_VMOD_DYNAMIC;
970 
971  if ( vmod == PRRE_VMOD_DYNAMIC )
972  ret = ret | PRRE_VMOD_DYNAMIC;
973 
974  if ( PRREhwInfo::get().getVideo().isDrawRangeElementsSupported() )
975  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_RNG_BIT);
976 
977  if ( PRREhwInfo::get().getVideo().isCompiledVertexArraySupported() )
978  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_CVA_BIT);
979 
980  // bForceUseClientMemory is considered only if modifying habit is dynamic
981  if ( bForceUseClientMemory && (vmod == PRRE_VMOD_DYNAMIC) )
982  return ret | BIT(PRRE_VT_VA_BIT); // client-side should use an array, even if none of RNG and CVA bits are available, to avoid selecting immediate mode
983 
984  // At this point we want to use host/server memory, but it could be still either client- or server-side array, or display list or immediate mode, so
985  // we should set PRRE_VT_VA_BIT only in case of some arraying.
986 
987  // if general VBO is supported, return with that immediately as we don't want to stick to any vendor-specific magical stuff.
988  if ( PRREhwInfo::get().getVideo().isVertexBufferObjectSupported() )
989  return ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT);
990 
991  if ( PRREhwInfo::get().getVideo().isATIVertexArrayObjectSupported() )
992  {
993  if ( PRREhwInfo::get().getVideo().isATIElementArraySupported() )
994  ret = ret | BIT(PRRE_VT_RNG_BIT) | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_ATI_2,PRRE_VT_VENDOR_BITS,3);
995  else
996  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_ATI_1,PRRE_VT_VENDOR_BITS,3);
997  }
998  else
999  {
1000  if ( PRREhwInfo::get().getVideo().isNVVertexArrayRangeSupported() )
1001  {
1002  if ( PRREhwInfo::get().getVideo().isNVElementArraySupported() )
1003  ret = ret | BIT(PRRE_VT_RNG_BIT) | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_NVIDIA_2,PRRE_VT_VENDOR_BITS,3);
1004  else
1005  ret = ret | BIT(PRRE_VT_VA_BIT) | BIT(PRRE_VT_SVA_BIT) | BITF_PREP(PRRE_VT_VENDOR_NVIDIA_1,PRRE_VT_VENDOR_BITS,3);
1006  }
1007  }
1008 
1009  // at this point, if VA_BIT is not set, the selected mode is either display list or immediate mode, depending on vmod
1010 
1011  return ret;
1012 } // selectVertexTransferMode()
